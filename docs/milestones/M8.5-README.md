# Milestone 8.5 — QChain × qstark integration

Wires the **qstark zk-STARK proof system** (M8.1–M8.4) into QChain's
anon-spend code path as a parallel "anon-stark" layer that coexists with the
existing M4 Schnorr-based anonymous transactions.

## What this gives you

A new transaction kind, `STARKAnonTransaction`, which proves:

> "I own one of the notes in the STARK pool. Here's my nullifier (so you can
> reject double-spends). Please credit `<recipient>` with `<amount>` of
> transparent value."

Crucially, the verifier learns nothing about *which* note was spent — that's
the actual anonymity property M4's Schnorr design couldn't give us (M4
reveals the spent leaf through its Merkle proof).

## What's in this milestone

```
qchain/crypto/anon_stark.py     # STARKNote + STARKAnonTree (Rp64_256 / Goldilocks)
qchain/chain/anon_stark_tx.py   # STARKAnonTransaction + create/verify
qchain/chain/blockchain.py      # additive: shield_to_stark_pool, submit_stark_anon
qchain/chain/block.py           # additive: stark_anon_transactions field
qchain/tests/test_anon_stark.py # 21 tests including the gap-documentation
```

The M4 anon code (`crypto/anon.py`, `chain/anon_tx.py`) is untouched. Both
paths work in the same chain, in the same block.

## Architecture decisions

### 1. Why parallel, not replacement?

The M4 Schnorr code path uses secp256k1 and SHA-256 commitments. The qstark
STARKs work in the Goldilocks field with Rp64_256 hashes. These are
**different cryptographic primitives over different fields**. Forcing all
M4 traffic through STARK would mean:

- Rewriting `crypto/commitments.py` and `crypto/merkle.py`
- Re-keying every wallet
- Losing the M4 multi-input / value-conservation properties that the
  current depth-4 STARK demo can't yet reproduce
- ~600 lines of Python rewrites plus reworking the 13 existing M4 tests

That's a real migration project. M8.5 instead adds a *new* path. Production
teams do this same thing for safe rollouts: ship v2 alongside v1, run them
in parallel, migrate when v2 is at feature parity.

### 2. Why split deposit (shield) from spend?

The blockchain exposes two separate operations:

```python
chain.shield_to_stark_pool(leaf_digest) → leaf_idx
chain.submit_stark_anon(stark_anon_tx)
```

A spend doesn't add new leaves. This matches how Tornado Cash and similar
production STARK-based pools are architected: `deposit` is its own operation
that appends a commitment; `withdraw` is the STARK-proved spend.

Why split them? Two reasons:

1. **AIR simplicity.** Modeling "spend AND create a change note" in one STARK
   would need to hash the new leaf inside the AIR (8 more rows), pushing the
   trace past the depth-4 layout that M8.3-full uses.
2. **Honest scope.** Chained spends are exactly where value-conservation
   cheating would hide. Splitting deposit from withdraw makes the trust
   surface explicit: the deposit operation carries a public `value`, and a
   real production hook would debit the depositor's transparent balance.

### 3. Why one input, no shielded outputs?

To keep the M8.5 STARK tractable. M4 supports multi-input, multi-output, and
shield/unshield in the same transaction (with Pedersen sums enforcing value
conservation). M8.5 supports exactly: **one leaf → unshield to a transparent
recipient + fee**. This is the minimum shape that demonstrates the
zk-Merkle-membership primitive.

## Known scope gaps (honestly stated)

These are real, and a production version would need to close them. Each is
**documented in the test suite** so it's harder to forget about.

### Gap A: Value conservation not enforced

The transaction carries a public `unshield_amount`. The STARK doesn't prove
that this amount equals the value committed in the leaf. A spender could
spend a note worth 50 while declaring `unshield_amount=10000`.

**Closing it:** put a range proof + value commitment inside the AIR, or
embed Pedersen-style value commitments and prove value balance algebraically.
Either way, it adds several hundred lines of AIR code and trace rows.

### Gap B: Nullifier not bound to the spent leaf

The published nullifier is a deterministic hash of the spender's secret key.
But the STARK *only* proves Merkle membership; it does NOT prove the
nullifier was correctly derived from the leaf being spent. An attacker who
knows a valid `(sk, leaf, path)` could publish a fresh nullifier each time
they spend, evading double-spend detection.

**Closing it:** the AIR must assert `nullifier = H(sk + 1, randomness, value)`
in the same proof that asserts `leaf = H(sk, randomness, value)` is in the
tree. The native Merkle/AIR support is already in qstark; this is a "new
proof statement" — not a hard cryptographic problem, just more work.

There's a test `test_known_gap_nullifier_not_bound_to_stark_proof` that
documents this gap by asserting the chain currently accepts a swapped
nullifier.

### Gap C: Anonymity set capped at 16

`MERKLE_DEPTH = 4` from qstark. Production wants 20+. Bumping requires
rebuilding the qstark crate with a larger constant and re-deriving the AIR
trace dimensions — straightforward but untested at depth >4.

### Gap D: Depositor identity revealed at shield time

`shield_to_stark_pool(leaf)` doesn't take a depositor address. A production
hook would couple it with a transparent debit of the shielder's balance.
Without that coupling, the chain doesn't enforce that the shielded value
came from anywhere — anyone can call shield with any leaf.

## What the test suite covers

21 tests, all passing:

**Note & tree basics** (8): leaf determinism, leaf distinguishes value,
nullifier determinism, nullifier domain-separation from leaf, nullifiers
distinguish secret keys, fresh tree roots independent of construction
state, root changes on append, capacity enforced, auth-path length.

**Happy path** (1): honest spend verifies, lands in block, mempool clears,
nullifier marked, recipient credited, proposer paid.

**Adversarial** (5): stale Merkle root rejected, double-spend (across
blocks) rejected, mempool double-spend rejected, tampered proof rejected,
negative amounts rejected, empty recipient rejected.

**Documented gap** (1): nullifier swap not detected (Gap B, see above).

**Coexistence** (1): M4 and M8.5 transactions land in the same block, both
verify, both credit correctly.

**Block compatibility** (1): a block with no stark transactions has the
same hash as it would have pre-M8.5 (no spurious payload changes).

**Serialization** (1): STARKAnonTransaction round-trips through to_dict/from_dict.

**Bytes round-trip** (1): digest bytes round-trip through the helpers.

**Tree capacity** (1): the 16-leaf limit raises ValueError.

**Tree determinism** (1): empty trees are bytewise equal.

## Real numbers

From running the full pipeline end-to-end:

| Operation | Time | Notes |
|-----------|------|-------|
| `shield_to_stark_pool(leaf)` | ~0.5 ms | 4 Rp64_256 hashes to update path to root |
| `create_stark_anon_tx(...)`  | ~3 ms   | dominated by STARK proving |
| `submit_stark_anon(stx)`     | ~0.3 ms | dominated by STARK verifying |
| Proof size                   | ~28 KB  | qstark M8.3 FULL output |

## Where this could go (M8.6+)

In rough order of value:

1. **Close Gap B (nullifier binding).** This is the most important
   correctness fix. Add the nullifier-derivation hash to the AIR. Probably
   1-2 weeks of focused work; mostly extending the M8.3-full trace layout.

2. **Close Gap A (value conservation).** Either Pedersen commitments inside
   the STARK, or range proofs over the leaf's value field. Larger project.

3. **Scale to production depth.** Rebuild qstark with `MERKLE_DEPTH = 20`
   or higher. Verify the AIR scales — should be a one-constant change but
   benchmarking matters.

4. **Couple shield-to-pool with a transparent debit.** Add a
   `shield_to_stark_pool(depositor_addr, value, leaf)` that requires a
   matching transparent debit. Closes Gap D.

The end state of those four changes is a production-grade shielded pool
with real anonymity, value conservation, and chain-level integrity. It's
the project that M8.5 sets up but deliberately doesn't try to complete in
one milestone.
