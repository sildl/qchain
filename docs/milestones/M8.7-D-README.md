# Milestone 8.7-D — Closing Gap D: on-chain shield transactions

This milestone closes **Gap D** from M8.5, which the M8.7 work surfaced
more clearly: the STARK pool wasn't chain-replicated. Each node held its
own pool, populated by direct `chain.shield_to_stark_pool(leaf)` calls
(originally from the dashboard).

After M8.7-D:
- Shielding is a **first-class on-chain transaction** (`ShieldTransaction`)
- It's **gossiped** to all peers
- It's **mined into blocks** like every other tx
- A fresh node that syncs the chain **deterministically reconstructs**
  the same STARK pool root
- The depositor is **debited** on the transparent ledger by the amount shielded

## What changed

### New type: `ShieldTransaction`

`qchain/chain/shield_tx.py` — a Dilithium-signed transaction carrying:
- `sender`: the depositor's address (overwritten by `sign()` from the keypair)
- `leaf`: the 4-tuple Goldilocks digest being added to the STARK pool
- `amount`: transparent coins to debit from the sender
- `timestamp`, `nonce`: standard replay protection
- `public_key`, `signature`: Dilithium

Verification rules:
1. Signature is valid
2. Sender address matches the public key (anti-impersonation)
3. Amount is positive
4. Leaf is a 4-tuple of non-negative u64

Wire format mirrors `Transaction`: newline-delimited JSON, base64
signature, leaf serialized as a list of four ints.

### Blockchain integration

`qchain/chain/blockchain.py`:
- `shield_mempool: List[ShieldTransaction]` added to state
- `submit_shield(shtx)` admits a shield to the mempool, with checks:
  - Signature verifies
  - Depositor has sufficient balance (accounting for already-pending
    shields and transparent txs from the same address — preventing
    same-mempool double-spend)
  - No duplicate txid in mempool
- `_apply_shield_tx(shtx)` appends the leaf to `stark_anon_tree`
- `balance_of(addr)` debits confirmed shield txs from the sender
- `mine_pending` and `propose_pending` both include shield txs and
  apply them in the correct order: **shields BEFORE STARK spends**
  so a same-block "shield then spend" pattern would work (in practice
  rare since the spender needs the leaf's pool index)
- `is_valid()` re-verifies shield tx signatures during chain replay

### Block layout

`qchain/chain/block.py`:
- New `shield_transactions` field
- Block header includes `shield_tx_root` (only if non-empty —
  preserves pre-M8.7 block hashes for backward compatibility)
- `to_dict` / `from_dict` round-trip the new field

### Network layer

`qchain/network/node.py`:
- New `new_shield_tx` message type (8 message types total now)
- `_handle_new_shield_tx` mirrors the anon/stark handlers: validate,
  dedup by txid, gossip on success, drop silently on `ValueError`
- `submit_shield_tx(shtx)` public method for local-submit-and-gossip
- `_try_extend_or_replace` extended:
  - **Direct extension**: applies shield txs before STARK spends in each
    newly-arrived block
  - **Fork resolution (candidate chain)**: now FULLY rebuilds
    `stark_anon_tree` from history (was preserving the existing tree
    in M8.7 because shields weren't on-chain; with M8.7-D they are)
  - **Orphaned blocks**: shield txs from dropped blocks are re-submitted
    to the mempool

### Dashboard

`qchain/dashboard/server.py`:
- `/api/stark/shield` now constructs a **signed** `ShieldTransaction`
  using the dashboard's `miner_wallet` as depositor. The shield goes
  to the mempool first; only when mined does the leaf appear in the pool.
- New `shield_mempool` field in the state snapshot
- New `shield_tx` WebSocket event type for live feed
- Wired `on_stark_anon_tx` and `on_shield_tx` callbacks so peer-side
  events also fire (previously stark-anon only fired on local submission)

The dashboard requires the miner to have mined enough blocks for the
shield to fit within their balance. The 400 response on insufficient
balance includes a hint to mine first.

## Honest scope notes

1. **Depositor identity is publicly visible.** The shield tx carries the
   depositor's address and amount. That's an intrinsic leak: somebody
   has to pay, so transparent-side accounting names them. Fully hiding
   deposits requires its own anonymous-deposit primitive (a mixer or
   CoinJoin layer) — out of scope.

2. **`is_valid()` doesn't re-verify STARK proofs.** Each STARK tx was
   verified at admission time, but chain replay (e.g., for fork
   resolution) re-validates only signatures, not STARK proofs. A
   production validator would do the latter, at the cost of ~0.4 ms
   per STARK tx in the chain.

3. **`is_valid()` doesn't re-verify balances for shield txs.** Same as
   for transparent txs — we trust `submit_shield` enforced this at
   admission. A full re-validation would replay all balances during
   chain walk.

4. **Same-block "shield then spend" is mechanically allowed but not
   practical.** The spender would need the leaf's pool index to
   construct the STARK proof, which they only know after the shield
   is applied. In practice shields land one block before spends.

5. **The dashboard's `pending_leaf_idx` is a best-guess.** Under
   concurrent shields it could be wrong (two parallel shields could
   both predict idx N). For a single-user demo this is fine. A real
   wallet would re-derive the index after observing the mined block.

## What's next

Gap A (value conservation via bit-decomposition range proofs) remains
the only major gap. See `M8.7-Gap-A-design.md` for the full design.
Roughly a week of work. After that, this is a serious end-to-end
shielded-payment system.
