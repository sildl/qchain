## What changed

### Part A — qstark_py bindings (`qstark_py/src/lib.rs`)

Two functions updated:

**`prove_m86_membership`** — old signature took `(sk, r, v, path, unshield, fee)`,
returned `(proof, root, nullifier)`. New signature takes
`(sk, r, v, path, unshield, fee, sk_out, r_out, v_out)` and returns
`(proof, root, nullifier, output_leaf)`. The witness-consistency
pre-check is now the M8.11 three-way equation
`v == unshield + fee + v_out` (was `v == unshield + fee`).

**`verify_m86_membership`** — old signature took
`(proof, root, nullifier, unshield, fee)`. New signature adds
`output_leaf`. All five public inputs are bound to the proof via
Fiat-Shamir.

The wheel was rebuilt with `maturin build --release` and reinstalled.
Existing `qstark_py/test_qstark_py.py` (21 tests for hash + M8.3
membership) still passes — those don't touch the M8.11 path.

### Part B — QChain chain integration

**`qchain/chain/anon_stark_tx.py`** (the core change):

- `STARKAnonTransaction` dataclass gained `output_leaf: Digest` field
- `txid()`, `to_dict()`, `from_dict()` updated to include output_leaf
  in the canonical encoding (so peers receiving via gossip see it)
- `verify()` passes output_leaf through to `q.verify_m86_membership`
- `create_stark_anon_tx` accepts new optional `change_note: STARKNote`
  parameter:
  - If provided, that's the spender's chosen change note. They retain
    `(sk_out, r_out, v_out)` privately and can spend it later.
  - If omitted, a random STARKNote with `value=0` is auto-generated
    (full-spend pattern with dummy output, matching Zcash Sapling)
- Validates `unshield_amount + fee + change_note.value == note.value`
- Validates the sum doesn't overflow u64
- Sanity checks: STARK's computed `output_leaf` matches `change_note.leaf()`

**`qchain/chain/blockchain.py`**:

- `_apply_stark_anon_tx` now appends `stx.output_leaf` to
  `stark_anon_tree` in addition to marking the nullifier
- `is_valid()` replay applies the same order during chain validation:
  shields first (extend pool), then STARK spends (mark nullifier +
  append output_leaf). Matches mine_pending exactly.

### Why dummy outputs for full spends

A natural-looking alternative was "if there's no change, skip the
output_leaf append." That's a privacy leak: anyone observing the chain
could identify full-spend transactions by the absence of an appended
leaf, defeating the unlinkability the project provides.

The Sapling-style fix: every spend appends a leaf, even if it's a
dummy with `value=0` and random `(sk_out, r_out)`. Real change notes
and dummies are computationally indistinguishable from the chain's
perspective. The cost is pool bloat — the STARK pool grows on every
spend, not just on shields — but that's acceptable for the privacy
gain.

### New integration tests (4, in `qchain/tests/test_anon_stark.py`)

| Test | What it proves |
|------|---------------|
| `test_m811_partial_spend_produces_change_note` | Partial spend leaves new leaf in pool; chain validates via is_valid() |
| `test_m811_change_note_can_be_spent_later` | Spender can spend the change note in a follow-up tx; both nullifiers recorded; balances correct |
| `test_m811_partial_spend_with_mismatched_value_rejected` | `unshield+fee+change_value ≠ note.value` rejected at construction with clear error |
| `test_m811_full_spend_still_works_with_default_change` | Backward-compat: omitting change_note still works, appends a dummy leaf |

The tests use the **shield-tx → mine → spend → mine** path, not the
test-shortcut `chain.shield_to_stark_pool` direct-append path. This is
required for `is_valid()` chain replay to work (a finding from M8.10:
direct-append bypasses the chain-state invariant). All four exercise
the M8.10 chain replay logic on chains containing partial-spend STARK
transactions.

## What Phase 3 does NOT do

- **Network-level adversarial coverage.** Tests for malicious peers
  tampering output_leaf, v_out, etc. between hops are Phase 4 work.
  The chain-side verify and is_valid both catch these, but the peer
  message handlers (`_handle_new_stark_anon_tx`) haven't been
  re-tested for the new field shape.
- **Wallet bookkeeping for tracking change notes.** When a spender
  creates a partial spend, they need to track the change_note and its
  eventual leaf index in the pool. The test
  `test_m811_change_note_can_be_spent_later` does this by scanning
  the pool for the matching leaf — works but isn't a real wallet
  API. A `Wallet.shielded_notes` collection or similar would be the
  next abstraction.
- **Dashboard UX for partial spends.** The "Spend STARK note" UI in
  the dashboard doesn't expose a v_out field. Users wanting to do
  partial spends from the dashboard can't yet. The Python API
  supports it; the UI doesn't surface it.
- **Persistence of change notes.** If a node restarts, the spender's
  knowledge of which leaves are their change notes is lost (along
  with everything else — the chain is in-memory). Same in-memory
  limitation that's been documented since the project began.

These are real concerns, the right scope notes to track. None of them
block Phase 3 from shipping; they're follow-on work.
