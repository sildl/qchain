# M8.9 — Sparse Merkle tree, depth 20

## What this is

The STARK pool and mixer pool both use a fixed-depth Merkle tree to
hold note commitments. The depth was 16 (65,536-leaf cap) since M8.7-C.
M8.9 bumps it to 20 (1,048,576-leaf cap) — matching the Tornado Cash
reference design — and switches both the Rust and Python tree
implementations from dense to sparse storage so the larger capacity
doesn't blow up memory or construction time.

## Why depth 20

The depth question deserved real analysis. Working from concrete
reference points:

- **Tornado Cash**: depth 20 (1M leaves) — the canonical privacy mixer
- **Zcash Sprout**: depth 29 (early design)
- **Zcash Sapling / Orchard**: depth 32 (current production)
- **Aztec data tree**: depth 32; nullifier tree: depth 256 (sparse)

For QChain specifically:

| Option | Capacity | Cost vs depth 16 | Verdict |
|---|---:|---|---|
| 20 | 1,048,576 | ~1.5× prove time | Matches Tornado, fits the threat model |
| 24 | 16,777,216 | ~1.7× | Arbitrary compromise |
| 28 | 268,435,456 | ~1.9× | Excessive |
| 32 | 4,294,967,296 | ~2.0× | Excessive for a research demo |

The deciding factor: **the bigger anonymity-set problem isn't capacity,
it's small real same-denomination simultaneous deposits.** Going to
depth 32 buys a 65,536× larger CEILING when the FLOOR (actual deposits
that overlap in time) is the constraint. Depth 20 lifts the cap to
"never going to run out for this demo's lifetime" without optimizing
the wrong axis.

## Why "sparse"

A dense implementation stores the full tree: 2^20 = 1,048,576 leaf
positions + ~1M parent nodes. At ~32 bytes per digest that's ~64MB
per tree, with ~2M hash operations to construct.

In Python that's seconds-per-tree initialization. In Rust still fine
but memory-heavy. The fix is **sparse storage**: only populated leaves
are kept; empty subtrees are represented by precomputed "zero subtree
hashes" at each level.

The trick: `hash_inner(0_leaf, 0_leaf) = some_constant_h1`. Similarly
`hash_inner(h1, h1) = some_constant_h2`. Build a fixed table of 21
zero-subtree hashes once (one per level) at process start. When a
tree needs the digest at `(level, idx)` and that position has never
been populated, look it up in the table.

Result: storage is O(populated_leaves × depth) instead of O(2^depth),
and construction is O(1).

## What changed

### Rust (`qstark/src/hash_air/merkle.rs`)

Rewrote `MerkleTree`:

- Storage: `HashMap<(usize, usize), Digest4>` for populated nodes only
- `zero_subtree_hashes()`: `OnceLock`-cached array indexed by level
- `set_leaf(idx, leaf)`: updates the leaf and propagates ancestors
  via the existing zero-subtree fallback for any missing siblings
- `from_leaves(leaves)`: now just calls `set_leaf` in a loop
- `node_at(level, idx)`: explicit accessor for single nodes
- `layer(level)`: materializes a full layer on demand (test-only;
  expensive at low levels of a depth-20 tree)
- Kept `auth_path`, `root`, `verify_path` API-compatible

### Rust (`qstark/src/hash_air/m86_native.rs`)

- `M86_MERKLE_DEPTH: 16 → 20`
- Fixed stale `M86_INIT_BLOCK_INDICES` constant (was `[0, 5]` from the
  depth-8 era; corrected to `[0, M86_MERKLE_DEPTH+1, M86_MERKLE_DEPTH+2]`)
- Updated docstring with depth-20 trace stats

**Critical AIR-side observation:** depths 16 AND 20 both fit in a 256-row
trace (active rows go 152 → 184 out of 256). Polynomial dimensions, FRI
parameters, and proof shape are unchanged. **Proof size is unchanged.**
This is what makes depth 20 affordable on the AIR side.

### Rust tests

5 test sites accessed `tree.layers[level][idx]` directly. Updated to
use `tree.root()` (when `idx >> DEPTH` is always 0) or `tree.node_at()`
(when the level matters). One legacy demo test (`m8_3_full_trace`)
correctly preserves its level-MERKLE_DEPTH partial-root semantics via
`node_at`.

### Python (`qchain/crypto/anon_stark.py`)

Rewrote `STARKAnonTree` with the same sparse approach:

- Storage: `Dict[Tuple[int, int], Digest]`
- `_zero_subtree_hashes()`: class-level cache, lazily initialized
- `append`, `root`, `auth_path` API-compatible
- New: `leaf_at(idx)` for single-leaf lookup
- **Compat shim**: `_SparseLayersView` provides
  `tree._layers[level][idx]` for legacy callers. Only single-element
  access supported; iteration/slicing intentionally unsupported.
  ~12 legacy callsites in the codebase use this pattern; the shim
  preserves them without rewriting.

### Python wheel

`qstark_py` rebuilt with depth 20. `q.m86_merkle_depth()` now returns
20. Confirmed end-to-end.

## What did NOT change

- The M4 anon pool's `MerkleTree` (`qchain/crypto/merkle.py`) — that
  uses BLAKE-based hashing for the Schnorr-ZK system, not the STARK
  pool. Still depth 16. Unrelated to M8.9.
- Block format
- Wire protocol
- Transaction structures
- Persistence schema (chain JSON, wallet JSON)
- AIR constraint composition

## Performance impact (measured)

| Op | depth 16 | depth 20 | ratio |
|---|---:|---:|---:|
| tree append | ~0.3 ms | 0.4 ms | 1.3× |
| auth_path | < 0.01 ms | 0.01 ms | n/m |
| prove m86 membership | ~25 ms | 37 ms | 1.5× |
| verify m86 membership | ~0.6 ms | 0.74 ms | 1.2× |
| **proof size** | 98,656 B | 97,732 B | **unchanged** |
| tree memory (empty tree) | ~64 MB | ~few KB | strictly better |
| tree __init__ time | ~3 s | < 1 ms | strictly better |

Proof size is unchanged because the trace length didn't change. Prove
and verify cost up modestly. Tree memory and construction time are
**dramatically better** — the sparse rewrite is what makes depth 20
affordable.

Zero new tests added in this pass — that's intentional. The existing
test suite IS the regression test for the depth change. Every
soundness test, integration test, and end-to-end test ran at depth
20 and produced the same outcome as at depth 16.

If proper depth-20-specific tests were added, they would verify:
- A leaf at index 1,000,000 (well past the depth-16 limit) is
  proof-verifiable
- Zero-subtree-hash caching is correct (no off-by-one in the level table)
- Sparse tree's `auth_path` for an unpopulated index produces a path
  whose verify against root succeeds (with leaf = zero digest)

These would be additive — the existing suite already proves AIR
correctness, integration with chain, network propagation, etc.

## Honest scope notes

- **Anonymity-set ceiling lifted, not the floor.** A 1M-capacity pool
  is meaningless privacy-wise unless many users deposit at the same
  denomination in the same window. The threat model entry T14 (timing
  analysis) is unchanged and remains the dominant privacy weakness.
- **No M8.9-specific tests added.** The reasoning is that existing
  tests fully exercise the AIR and integration; depth is just a
  constant. A more rigorous pass would add a "spend leaf at idx
  >65,536" test to explicitly exercise the new capacity range. Worth
  doing if you want defense-in-depth on the depth change.
- **Stale comment** in m86_native.rs about `M86_INIT_BLOCK_INDICES`
  was fixed as a bonus. It's an unused constant; the fix is for code
  hygiene only.
- **Python `_SparseLayersView` shim is minimal.** Only `__getitem__`
  works. If any code reaches into `_layers` and does `len()`, slice,
  iterate, etc., it will fail. None of the current code does that;
  the shim is just enough for the existing call sites.
- **Performance regressions are honest.** Prove time up 1.5×, verify
  up 1.2×. Acceptable for a research project; production might want
  to revisit if proving throughput becomes a bottleneck.

## Why this was the right milestone choice

M8.9 was the last Tier-2 item from the original tier analysis. It
delivers:

- Real cryptographic milestone (depth ↑, capacity ↑ by 16×)
- Production-grade design pattern (sparse Merkle is what Tornado /
  Aztec / Zcash all use)
- No protocol changes (back-compatible)
- No new attack surface (same AIR, same constraints)
- Honest performance numbers

The roadmap entry envisioned a fancier construction. The actual
implementation is mundane in the best way: read the references,
pick the right depth, write the sparse storage properly, verify
nothing regressed.
