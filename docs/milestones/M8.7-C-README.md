# Milestone 8.7-C — Fully closing Gap C: 65,536-note anonymity set

This milestone fully closes **Gap C** from M8.5. Originally listed as
"anonymity set ≤ 16," then partially closed in M8.7 to 256 (depth 8),
M8.7-C lifts the anonymity set to **65,536 notes** (depth 16) — a 4096×
increase over the original M8.5 limit, and a 256× increase over M8.7.

## What was blocking it

After M8.7, two constants needed to match for soundness:

- `M86_MERKLE_DEPTH` in `qstark/src/hash_air/m86_native.rs` — the AIR's
  expected path length
- `TREE_DEPTH` in `qstark/src/hash_air/merkle.rs` — the test-side
  MerkleTree helper's depth

The AIR was depth-parametric and was already verified working at depth
12 via the synthetic-witness bench. But `TREE_DEPTH` was hardcoded at 8,
so the smoke and soundness tests — which use `MerkleTree::from_leaves`
to build real witness paths — couldn't run at higher depths.

**The required invariant: `TREE_DEPTH == M86_MERKLE_DEPTH`.** If
TREE_DEPTH > MERKLE_DEPTH, the proof attests against an intermediate
hash (not the tree's actual root). If TREE_DEPTH < MERKLE_DEPTH, the
auth path is too short and the AIR's length assertion fails.

## What changed

Two lines:

```rust
// qstark/src/hash_air/merkle.rs
pub const TREE_DEPTH: usize = 16;  // was 8

// qstark/src/hash_air/m86_native.rs
pub const M86_MERKLE_DEPTH: usize = 16;  // was 8
```

That's the entire functional change. Everything else (AIR, trace
generation, periodic columns, constraint counts) is derived from these
constants and updated automatically.

Plus rebuilding the `qstark_py` wheel so `q.m86_merkle_depth()` reports
the new value to QChain. QChain reads this at module load time and
adapts `ANONYMITY_SET_SIZE = 1 << MERKLE_DEPTH` automatically.

## Why depth 16 (and not 20)

I weighed three options:

| Depth | Set size | Mem/tree | Proving | Notes |
|------:|---------:|---------:|--------:|-------|
| 12    | 4,096    | 256 KB   | 5.9 ms  | Bench-verified earlier |
| **16** | **65,536** | **4 MB** | **13.8 ms** | **Shipped** |
| 20    | 1M       | 64 MB    | ~25-50 ms* | Memory-hungry, slow tests |
| 24    | 16M      | 1 GB     | ~50-100 ms* | Not practical without sparse tree |

*estimated

Depth 16 was the sweet spot: large enough to be meaningfully
"production-shape" (Tornado Cash's original mixer used depth 20-32, but
65K is a strong anonymity set by any standard), with memory and proving
costs that stay comfortable.

Going to depth 20 would require either (a) accepting 64 MB per tree
allocation, or (b) implementing a sparse-tree optimization (recognize
zero-padded subtrees and cache their digests). Option (b) is a
worthwhile future refactor but wasn't necessary for this milestone.

## Real measurements

Single-core, release-mode bench at each depth:

| Depth | Set size | Active rows | Trace len | Prove (ms) | Verify (ms) | Proof (KB) |
|------:|---------:|------------:|----------:|-----------:|------------:|-----------:|
|     4 | 16       | 48          | 64        | 3.25       | 0.345       | 28.3       |
|     8 | 256      | 80          | 128       | 6.62       | 0.399       | 32.8       |
|    12 | 4,096    | 112         | 128       | 5.92       | 0.313       | 33.3       |
|  **16** | **65,536** | **144**       | **256**       | **13.83**      | **0.454**       | **38.1**       |

Note the scaling: depth 4→8 doubled prove time (16× more notes, 2× time).
Depth 8→12 actually got faster (same padded trace length).
Depth 12→16 doubled prove time again (trace went from 128 to 256 rows).

The proof size grew modestly — about 10 KB across the whole range from
depth 4 to 16. The verifier stayed under half a millisecond. **The dominant
cost is trace length, not depth per se.**

## Honest scope notes

1. **Memory per tree.** At depth 16 each `MerkleTree` instance holds
   2*65,536 = 131,072 digests = ~4 MB. The test suite creates many
   trees; total transient memory is maybe 100-200 MB during a full
   pytest run. Fine on any reasonable dev machine, but worth noting.

2. **Tree construction time.** The naive `from_leaves` pads to NUM_LEAVES
   with `ZERO_DIGEST` and hashes layer by layer. At depth 16 that's
   ~130K Rescue-Prime hashes per tree (~0.5s on this machine). For
   tests this is acceptable. For production a sparse Merkle tree
   (Zcash-style) would amortize this — empty subtrees have a
   deterministic digest per level, computable once.

3. **The bench at depth 20 was demonstrated but not shipped.** The AIR
   itself worked at depth 12+ (bench passed at 4096-leaf sets). Going
   to 20 would primarily require the sparse-tree optimization to keep
   tests fast and memory bounded.

4. **No other QChain code changed.** Everything reads `MERKLE_DEPTH`
   via `qstark_py.m86_merkle_depth()` at module load, so the depth bump
   propagates automatically. The dashboard, network layer, and tx
   validation all just work at the new depth.
