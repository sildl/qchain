### Defense layers (post-Phase 3)

For any STARK-anon spend, Gap A is now enforced by THREE independent layers:

1. **Construction time** (`create_stark_anon_tx`):
   - `value conservation`: unshield_amount + fee must equal note.value
   - `overflows u64`: u64-overflow defense
   - These fire BEFORE proof generation, with clear errors

2. **AIR / proof generation** (build_m86_trace):
   - `witness inconsistency`: v ≠ unshield_amount + fee in field arithmetic
   - Binary + decomposition constraints prove v ∈ [0, 2^64)

3. **Verification / submit time** (submit_stark_anon → verify_m86_membership):
   - FS-bound public inputs (root, nullifier, amount, fee) must match what
     the proof was generated for — any tampering invalidates the proof

An attacker would have to break all three to succeed. Each independently
catches misuse.

### Surprising finding: Fiat-Shamir binds individual values

I initially expected `swap_amount_and_fee_with_same_sum` to be ACCEPTED
by the AIR, because the only AIR constraint involving amount and fee is
`amount + fee == v` — symmetric in amount and fee.

It's REJECTED in practice because Winterfell's Fiat-Shamir transcript
includes the public inputs in fixed order (`root`, `nullifier`, then
`amount`, then `fee`). Changing the partition changes the FS challenge
sequence, which invalidates the proof.

This is a useful protocol-level property: even an honest prover can't
trick the chain into recording a different fee partition than what
they generated the proof for. The AIR is symmetric in amount/fee, but
the proof system as a whole is not.

### Field-wrap attack: AIR can't defend, chain must

One test (`field_wrap_attack_documented_as_chain_layer_concern`)
explicitly documents an attack the AIR doesn't defend against:

The AIR asserts `state[6] == Felt::new(amount) + Felt::new(fee)` —
a field-arithmetic equation. If a malicious chain accepted
`(amount, fee)` where `amount + fee` overflows u64 but `(amount + fee)
mod p` equals a valid u64 representable in leaf v, the AIR would
verify.

Phase 3 chain integration must reject any STARK tx where
`(amount as u128) + (fee as u128) > u64::MAX`. Easy to enforce in
Python. The test passes by definition — it documents the attack
surface at the AIR/chain boundary.

## Why these design choices

**Why bits as separate columns (not derived):**
Decomposing v into bits requires the prover to commit to a specific bit
representation. Putting bits in trace columns lets the constraints
inspect them. Without trace presence, the verifier has no way to check
the decomposition.

**Why binary constraints on every active row, not just row 0:**
Cheaper to express in Winterfell's periodic-column model. The bits are
constant across rows (filled identically); checking binary on every
row catches any tampering. The decomposition only needs row 0 because
the bit columns are constant.

**Why bits NOT constrained constant between rows:**
Not needed for soundness. The decomposition at row 0 + binary checks on
all rows are jointly sufficient to prove `0 ≤ v < 2^64`. Adding a
constancy constraint would cost 64 more transition constraints with no
soundness gain.

**Why `amount_plus_fee` stored in `M86Air`:**
The assertion is `state[6][0] == unshield_amount + fee` — that's a
public-value boundary assertion. The Air needs the value at
construction time (computed once from `M86Inputs`).

**Why bind to `state[6]` (the leaf-block's `v` cell) instead of `COL_V` (col 19):**
At row 0, the leaf-block-binding constraint already enforces
`state[6] == COL_V`. So they're equal at row 0. State[6] is the more
direct cell since it's the value going into the leaf hash. Either
choice works; state[6] is slightly cleaner.

## Soundness story so far

**What's proven by the AIR after Phase 1:**

For any accepted proof, there exists a witness `(sk, r, v, path,
unshield_amount, fee, b[0..64])` such that:

1. `leaf = H(sk, r, v)` (existing M8.6)
2. `leaf` is at the claimed Merkle position with the claimed path (existing M8.6)
3. `root` of the proof matches the chain's STARK pool root (existing M8.6)
4. `nullifier = H(sk+1, r, v)` (existing M8.6 / Gap B)
5. **NEW**: `b[i] ∈ {0, 1}` for all i in 0..64
6. **NEW**: `v = Σ b[i] * 2^i`
7. **NEW**: `v = unshield_amount + fee` (field arithmetic)

(5) + (6) jointly prove `v ∈ [0, 2^64)`. Combined with (7), the chain
knows: there's a leaf in the pool with the spender's secret key, and
that leaf's value (range-proved to be a u64) equals the publicly
declared unshield+fee amount.

**What's NOT yet enforced (deferred to Phase 3 chain integration):**

- The `unshield_amount` and `fee` in `M86Inputs` must match the
  `unshield_amount` and `fee` in the corresponding `STARKAnonTransaction`
  fields. Right now the proof and the transaction body are independent
  data structures — the chain doesn't yet cross-check them.

- Field-wrap attacks at the `unshield_amount + fee` boundary. If a
  malicious user submits `unshield_amount = 2^63, fee = 2^63 + 5`, the
  field sum is 5 (mod p ≈ 2^64), and the bit decomposition would prove
  `v = 5`. They'd extract 5 coins from a pool note worth 5, technically
  consistent. The fix: chain rejects any tx where the u64 sum overflows.
  Easy to enforce in Python (Phase 3); also doable in the AIR with
  another range proof on `unshield_amount + fee` itself, but unnecessary
  complexity if the chain enforces.

### The headline test in detail

`test_phase4_malicious_peer_tampered_amount_rejected_remotely` models the
worst case: a malicious node that intentionally skips its own outbound
validation. The test:

1. Sets up two connected nodes
2. Funds a depositor wallet and shields a 100-coin note (both nodes converge)
3. The malicious n1 builds an HONEST proof for `unshield_amount=100`
4. Post-construction, n1 tampers `stx.unshield_amount = 50` — trying to
   record only 50 in the chain ledger while consuming a 100-coin leaf
5. n1 bypasses its own `submit_stark_anon_tx` (which would catch the
   tampering) by calling `n1._gossip(...)` directly with the tampered
   payload
6. n2 receives the gossip, runs verification: the proof's Fiat-Shamir
   transcript was bound to amount=100, but the public input in the
   payload says 50, so FS challenges mismatch and verify returns false
7. n2's mempool stays empty; the attack fails

Without Phase 3's cross-check, this attack would succeed and n2 would
record an underspend in its mempool while the network as a whole would
consume a 100-coin leaf for a 50-coin unshield — silent value
destruction.

### Defense layers (final, post-Phase 4)

Gap A is now blocked at **four** independent layers:

1. **Construction time** (`create_stark_anon_tx`): value conservation +
   u64-overflow checks; raises ValueError before any proof generation
2. **AIR / proof generation** (`build_m86_trace`): witness-consistency
   assertion + bit-decomposition + binary constraints
3. **Local submission** (`submit_stark_anon` → `verify_m86_membership`):
   FS-bound public inputs cross-check
4. **Network propagation** (peer's `_handle_new_stark_anon_tx`):
   identical cross-check on receipt; tampering between hops detected

An attacker needs to break all four. (1) and (3)+(4) are application-level
guards; (2) is the cryptographic core that makes (3)+(4) sound.
