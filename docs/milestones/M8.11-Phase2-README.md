## What Phase 2 verified

### Structural — Winterfell accepts the new AIR

The first thing Phase 2 needed to confirm was that Winterfell would
accept the refactored AIR shape. If the constraint degree specs were
wrong, or the periodic-column counts were off, the prover or verifier
would panic before any soundness test could run.

```
cargo test --release --test m86_smoke
# PASS — proof generated (98,875 bytes), verified.
```

No degree-spec mismatches surfaced. The constraint registration in
M86Air::new is consistent with what evaluate_transition computes.

### Regression — M8.6 + M8.8-A1 soundness preserved

The big risk in any AIR refactor is silently breaking existing
soundness. Phase 2 ran the full M8.6 and M8.8-A1 adversarial suites
against the M8.11 AIR:

```
cargo test --release --test m86_soundness
# PASS — 10 tests (Gap B nullifier-binding):
#   rejects_swapped_nullifier_completely
#   rejects_wrong_root
#   rejects_wrong_nullifier
#   many_random_tampers_all_rejected (30 byte flips)
#   ... 6 more

cargo test --release --test m86_gap_a_soundness
# PASS — 14 tests (Gap A range proof + value binding):
#   wrong_unshield_amount_in_public_inputs_rejected
#   wrong_fee_in_public_inputs_rejected
#   swap_amount_and_fee_with_same_sum_rejected_via_fiat_shamir
#   field_wrap_attack_documented_as_chain_layer_concern
#   ... 10 more
```

24 prior adversarial tests still pass. No M8.6 or M8.8-A1 property
was broken by the M8.11 AIR refactor.

| Category | Tests | What's proven |

### What this proves about M8.11 soundness

Together, the 28 adversarial tests (14 Gap A + 14 new partial-spend)
plus 10 Gap B tests plus diag green plus an honest smoke prove+verify
constitute strong evidence that the M8.11 AIR enforces:

- **Membership**: the input leaf is in the pool (M8.6 property)
- **Ownership**: only the spender knows (sk, r) for that leaf (M8.6)
- **Single-use**: the nullifier prevents double-spend (M8.6)
- **Value binding (M8.8-A1)**: v_in ∈ [0, 2^64) AND v_in = unshield + fee + v_out (extended in M8.11)
- **Change-note integrity (M8.11)**: the public output_leaf equals H(sk_out, r_out, v_out) for the spender-chosen secrets
- **Range proof on v_out (M8.11)**: v_out ∈ [0, 2^64) preventing field-wrap siphoning
- **All bindings via Fiat-Shamir**: tampering ANY public input after proof generation invalidates verification

## Performance — M8.11 vs M8.10

Measured at depth 16 (65,536-note anonymity set), single core, release mode:

| Metric | M8.10 | M8.11 | Change |
|--------|------:|------:|-------:|
| Proving time | 22 ms | **30 ms** | +38% |
| Verifying time | 0.6 ms | **0.55 ms** | ~flat |
| Proof size | 66 KB | **97 KB** | +47% |
| Trace width | 84 cols | **151 cols** | +80% |
| Active rows | 144 | **152** | +6% |
| Constraints | 113 | **194** | +72% |

The prover pays for the v_out bit columns; the verifier essentially
doesn't (FRI verification scales with the log of the trace length,
not the column count). This is the right tradeoff — verifiers run
on every node.

Total qstark test count: **96 → 110** (+14 new M8.11 tests). All green.

## Honest scope notes — what Phase 2 does NOT prove

Phase 2 proves the M8.11 AIR has soundness equivalent to M8.6 +
M8.8-A1, extended with change-output integrity. It does NOT prove:

- **Chain-level soundness for partial spends.** The chain still
  doesn't know about output_leaf. `STARKAnonTransaction` doesn't
  carry a change-note field, `submit_stark_anon` doesn't check
  one, `mine_pending` doesn't append the new leaf to the STARK
  pool. Phase 3 work.
- **Network-level adversarials.** A malicious peer that tampers
  output_leaf between hops. Phase 4 work.
- **Wallet-level concerns.** The spender needs to track which
  leaf index their change note lands at, similar to the
  pending_leaf_idx race documented in M8.7-D. Out of scope for
  the cryptographic core.

## Phase 2's surprise finding

When running the existing M8.8-A1 adversarials against the M8.11 AIR,
I expected some to fail (because the AIR shape changed substantially).
None did. This is evidence that the M8.11 refactor was clean — every
M8.8-A1 constraint group survived intact, including the Group 10
value-conservation constraint that *replaced* the M8.8-A1 amount-sum
boundary assertion. Tests that verified the amount-sum binding via
"wrong unshield_amount rejected" still pass, because Group 10
enforces the same binding via a slightly different mechanism.

This was a real risk that didn't materialize. Worth documenting because
the *next* AIR refactor (M8.11 Phase 3 or M9) needs to maintain the
same property.
