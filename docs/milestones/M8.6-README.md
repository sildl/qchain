# Milestone 8.6 — Closing Gap B: nullifier-bound STARKs

The M8.5 README listed four known gaps. M8.6 closes **Gap B** — the most
important one for soundness — by extending the STARK to prove the nullifier
is correctly derived from the spent leaf.

## The problem (M8.5)

In M8.5, the STARK proved Merkle membership: "I know a leaf at some position
in the tree." But the nullifier was *separate*. The chain trusted the spender
to compute it correctly. An attacker holding a valid `(sk, r, v, path)` could
publish ANY nullifier, including a fresh one, to spend the same note multiple
times.

The M8.5 test suite had `test_known_gap_nullifier_not_bound_to_stark_proof`
that explicitly demonstrated this attack succeeding.

## The fix (M8.6)

Extend the AIR so the same proof asserts:
1. `leaf = H(sk, r, v)`
2. `leaf` is in the Merkle tree with public root `R`
3. `nullifier = H(sk + 1, r, v)`

Public inputs: `(root, nullifier)`. Witness: `(sk, r, v, path)`.

If the spender tries to swap the nullifier, the proof's *internally
computed* nullifier won't match the *claimed* one — and verification fails.

## What changed

### Rust crate (`qstark/`)

Two new files:
- `src/hash_air/m86_native.rs` — reference implementation: leaf hash, 4-level
  Merkle climb, nullifier hash. Used by the diagnostic test.
- `src/hash_air/m86_air.rs` — the AIR with 6 hash blocks (1 leaf + 4 Merkle
  + 1 nullifier) over a 64-row, 20-column trace.

The AIR has 48 transition constraints in 7 groups:
1. **Hash round** (12, deg 8) — same as M8.3
2. **Merge boundary** (12, deg 4) — M8.3's swap logic, gated to skip the
   nullifier-block boundary
3. **Nullifier boundary** (12, deg 3) — NEW: at row 39→40, force the next
   block's state to `(3, 0, 0, 0, sk+1, r, v, 0, 0, 0, 0, 0)`
4. **Direction bit binary** (1, deg 2) — same as M8.3
5. **Within-block static** (5, deg 2) — same as M8.3
6. **Global witness static** (3, deg 2) — NEW: `sk`, `r`, `v` are carried in
   3 columns and must stay constant across the whole active trace
7. **Leaf-block binding** (3, deg 2) — NEW: at row 0, state[4..7] must equal
   the witness columns for `(sk, r, v)`

Three new periodic columns: `is_active`, `is_nullifier_boundary`, `is_row_0`.

### Python bindings (`qstark_py/`)

Two new functions:
```python
proof, root, nullifier = qstark_py.prove_m86_membership(sk, r, v, path)
ok = qstark_py.verify_m86_membership(proof, root, nullifier)
```

### QChain (`qchain/`)

`crypto/anon_stark.py` and `chain/anon_stark_tx.py` migrated to use the M8.6
proofs. Same on-chain wire format — only the proving/verifying API calls
changed. The M8.5 "known gap" test was renamed to
`test_m86_closes_gap_b_swapped_nullifier_rejected` and now asserts that the
nullifier swap is REJECTED.

### Dashboard (`qchain/dashboard/`)

Now visualises the STARK pool alongside the M4 anon pool:
- **Stats grid**: STARK pool size + STARK nullifiers (cyan)
- **Mempools**: third column for STARK-anon transactions
- **Block list**: shows `Nt + Na + Ns` counts per block
- **Controls panel**: "Shield to STARK pool" + "Spend STARK note" buttons
- **Live events feed**: handles the new `stark_anon_tx` event type with
  cyan styling
- **Two new API endpoints**: `POST /api/stark/shield`, `POST /api/stark/spend`

## Real numbers

Measured from Python, single core, M8.6 happy path:

| Operation | Time | Size |
|-----------|------|------|
| Trace generation     | ~0.5 ms  | 20 cols × 64 rows |
| `prove_m86_membership` | ~3.4 ms | 29,165 byte proof |
| `verify_m86_membership` | ~0.4 ms | — |

Compared to M8.3-FULL: proving is ~20% slower (more constraints to evaluate),
proof size grew ~5% (more constraints to commit to). For closing the central
correctness gap, this is a cheap upgrade.

## Remaining gaps (from M8.5 README, status update)

| Gap | M8.5 | M8.6 |
|-----|------|------|
| A — Value conservation not enforced | open | open (no change) |
| **B — Nullifier not bound to leaf** | open | **CLOSED** |
| C — Anonymity set capped at 16 | open | open (qstark depth still 4) |
| D — Depositor identity not coupled | open | open (no change) |

## What's next

Natural progressions in order of impact:

1. **Bump anonymity set (Gap C).** Rebuild qstark with `MERKLE_DEPTH = 20`
   or higher and re-derive the AIR's trace dimensions. Mostly a constants
   change, but worth benchmarking proving time scaling.

2. **Close Gap A (value conservation).** Add range proofs + value
   commitments inside the AIR. Several hundred lines of additional AIR
   code; the harder remaining gap.

3. **Network-propagate STARK transactions.** The p2p layer doesn't yet
   gossip `stark_anon_transactions`. Easy to add: a `new_stark_anon_tx`
   message kind in `network/node.py`.

4. **Wallet integration.** Currently the dashboard tracks owned STARK
   notes in memory only. A real wallet would persist them and offer
   change-address management.
