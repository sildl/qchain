# M8.7 — Gap A Design Sketch: Value Conservation via Range Proofs

This is a **design document, not an implementation.** Gap A closure is its own
milestone, scoped roughly equivalent to M8.5 + M8.6 combined.

## Why it matters

M8.6 closed Gap B: the nullifier is now bound to the leaf via the STARK.
But the **declared unshield amount** in the transaction is still
unconstrained by the leaf's actual value.

Today, an attacker holding a valid `(sk, r, v=10, path)` could:
1. Generate a valid M8.6 proof for spending the leaf
2. Submit a transaction with `unshield_amount = 1_000_000`
3. The chain accepts it because the STARK doesn't check value

This is the central reason M8.5/M8.6 is a **demo**, not production.

## The technique: bit-decomposition range proof

The standard way to enforce `v == unshield_amount + fee` inside a STARK
without revealing `v` is:

### New witness columns

Add 64 new columns `b[0..64]` to the AIR trace:
- `b[i] = (v >> i) & 1` — the i-th bit of v
- The prover supplies these as part of the witness

### New transition constraints

**Binary constraints (64 of them, degree 2):**
```
for i in 0..64:
    b[i] × (b[i] - 1) == 0   // enforced on all active rows
```

This proves each `b[i] ∈ {0, 1}`. Combined with decomposition, it
proves `v < 2^64`.

**Decomposition constraint (1, degree 1):**
```
at row 0:  v == Σ_{i=0..63} b[i] × 2^i
```

Pairs with binary constraint to prove `v` decomposes correctly.

**Value binding constraint (1, degree 1):**
```
at row 0:  v == unshield_amount + fee
```

Where `unshield_amount` and `fee` are NEW public inputs. This binds
the leaf's value to what's declared on-chain.

### Public inputs (expanded)

```
M86_v2_PublicInputs:
  root             : Digest4       // existing
  nullifier        : Digest4       // existing (M8.6)
  unshield_amount  : u64           // NEW (Gap A)
  fee              : u64           // NEW (Gap A)
```

## Architectural impact

### Trace growth

| | M8.6 | M8.7 (proposed) |
|---|---|---|
| Columns | 20 | 84 (20 + 64 bit columns) |
| Rows | 128 (depth 8) | 128 (same) |
| Cells | 2,560 | 10,752 (~4× memory) |

### Constraint count

| | M8.6 | M8.7 |
|---|---|---|
| Hash round | 12 | 12 |
| Merge boundary | 12 | 12 |
| Nullifier boundary | 12 | 12 |
| Direction binary | 1 | 1 |
| Within-block static | 5 | 5 |
| Global witness static | 3 | 3 |
| Leaf-block binding | 3 | 3 |
| **Bit binary** | — | **64** |
| **Value decomposition** | — | **1** |
| **Value binding** | — | **1** |
| **Total** | **48** | **114** |

### Performance estimate

Working hypothesis: STARK proving time is roughly linear in `cells × constraints`.
The bit-decomposition constraints are degree 2, same as existing binary
constraints, so they shouldn't blow up the constraint degree (which would
require more FRI work). Best estimate:

| | M8.6 (depth 8) | M8.7 (depth 8 + Gap A) |
|---|---|---|
| Proving time | 6.6 ms | ~15-25 ms |
| Proof size | 32.8 KB | ~38-45 KB |
| Verifying time | 0.4 ms | ~0.5 ms |

These are educated guesses; would need benchmarking.

## Implementation sketch

### Trace builder

```rust
// In build_m87_trace, after filling hash columns:
let v_u64: u64 = w.v.as_int();
for i in 0..64 {
    let bit = ((v_u64 >> i) & 1) as u64;
    for row in 0..M86_ACTIVE_ROWS {
        trace[20 + i][row] = BaseElement::new(bit);
    }
}
```

### Binary constraint group

```rust
fn evaluate_bit_binary<E: FieldElement>(
    frame: &EvaluationFrame<E>,
    is_active: E,
    result: &mut [E],
) {
    let current = frame.current();
    for i in 0..64 {
        let b = current[20 + i];
        result[i] = is_active * b * (b - E::ONE);
    }
}
```

### Decomposition constraint

```rust
fn evaluate_value_decomposition<E: FieldElement>(
    frame: &EvaluationFrame<E>,
    is_row_0: E,
    result: &mut [E],
) {
    let current = frame.current();
    let v = current[14];  // value column from M8.6
    let mut sum = E::ZERO;
    let mut pow = E::ONE;
    for i in 0..64 {
        sum += current[20 + i] * pow;
        pow *= E::from(2u32);
    }
    result[0] = is_row_0 * (v - sum);
}
```

### Binding constraint (assertion)

```rust
// In get_assertions:
let v_from_inputs = BaseElement::new(
    self.unshield_amount + self.fee
);
assertions.push(Assertion::single(14, 0, v_from_inputs));
```

Wait — this leaks v. Actually we'd want a transition constraint that
binds them WITHOUT publishing v separately. Since the trace already
constrains `v` to decompose into known bits whose sum equals
`unshield_amount + fee` (both public), we get the binding from the
combination of (decomposition + assertion on the sum).

Actually the cleanest way: since `unshield_amount + fee` is public,
just assert that `Σ b[i] × 2^i == unshield_amount + fee` directly
(without involving v as a separate column). Then `v` is *derived*
from the bits.

That's actually a simpler design. Let me restate:

**Simpler design:**
- Remove `v` from the global-witness-static columns; instead, derive `v`
  from the bit columns as `v = Σ b[i] × 2^i`.
- The leaf-block binding constraint becomes: at row 0, `state[6] == Σ b[i] × 2^i`.
- The transition's `unshield_amount + fee` assertion becomes:
  `Σ b[i] × 2^i == unshield_amount + fee` (as a boundary assertion at row 0).

This removes one witness column (no global v), saves a constraint group
(global witness static for v), and is cleaner.

## Testing strategy

A range-proof milestone needs:

### Soundness tests (must all reject)
- **Overflow attack**: claim `unshield_amount = 2^64 - 5` when `v = 5`
- **Bit forgery**: set some `b[i] = 2` (not 0 or 1) and try to match
- **Decomposition mismatch**: bits sum to a different value than `v`
- **Tampered amount**: swap `unshield_amount` in public inputs
- **Tampered fee**: swap `fee` in public inputs
- **Mass-overflow**: try to mint coins by claiming high amount on low-v leaf
- **Negative amounts**: try `unshield_amount` near field modulus
  (modular wrap-around — must be rejected by range proof)

### Happy path
- Various (v, unshield_amount, fee) combinations
- Boundary cases: `v=0`, `v=2^64-1`, `unshield_amount=v` (zero fee),
  `fee=v` (zero unshield), partial spends with change output

### Integration
- End-to-end QChain flow with the new constraint
- Block validation rejects STARK txs with `proof_amount != tx_amount`
- Mempool admission checks both proof validity and amount consistency

## Honest scope

This milestone is roughly equivalent to **M8.5 + M8.6 combined**:

- M8.5 was 4 days of work (designing and integrating STARK txs)
- M8.6 was 2 days (adding nullifier binding)
- Gap A would be **~5-7 days**: adding 66 constraints + 64 columns,
  refactoring witness, comprehensive soundness tests, performance
  measurement.

Not a session-sized milestone.

## What this doesn't solve

Even after Gap A closure, the design still has:

- **Gap C** — anonymity set capped by `M86_MERKLE_DEPTH`. Currently 256
  at depth 8; could go to 4096+ if the test MerkleTree helper is
  also expanded. Depth 12 was benchmarked and works mathematically;
  shipping it needs test infrastructure work.

- **Gap D** — STARK pool isn't chain-replicated. Each node maintains
  its own pool via dashboard shielding. For real multi-node STARK
  usage you'd add a `ShieldTransaction` type that's signed by the
  depositor and gossipped like any other tx, debiting a transparent
  balance and adding the resulting leaf.

These are separate milestones in their own right.

## Recommendation

**If pursuing:** budget a full week (~5 working days):
- Day 1: AIR refactor — add bit columns, binary constraints, decomposition
- Day 2: Integration with leaf/nullifier blocks + boundary assertions
- Day 3: Soundness tests (target 15+ adversarial cases)
- Day 4: Performance measurement + tuning
- Day 5: QChain integration + tx body changes + dashboard updates

**Alternative:** consider closing Gap D first (chain-replicated STARK pool)
since that gates real multi-node usage and is simpler — ~2 days.
A range-proof milestone on top of Gap D would yield production-realistic
shielded payments. Without Gap D, Gap A closure helps but the pool
is still single-node.
