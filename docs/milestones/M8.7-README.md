# Milestone 8.7 — P2P STARK propagation + depth bump

This milestone closed two of the three remaining gaps:
- **P2P propagation of STARK transactions** — STARK txs now gossip
  across the network like M4 anon txs.
- **Anonymity-set bump** — `M86_MERKLE_DEPTH` raised from 4 to 8,
  growing the anonymity set from **16 to 256 notes** with only a 2×
  proving-time cost.

Gap A (value conservation) remains open — see
`M8.7-Gap-A-design.md` for the full design and why it's a separate
milestone.

## Part 1: P2P propagation

### What changed

`network/node.py` extended:
- Imported `STARKAnonTransaction`
- Added `on_stark_anon_tx` callback hook in `__init__`
- Added `_handle_new_stark_anon_tx` message handler — mirrors
  `_handle_new_anon_tx` exactly
- Added `submit_stark_anon_tx` method — local submit + gossip
- Added `new_stark_anon_tx` to message dispatch
- Updated `_try_extend_or_replace` to apply STARK txs from new blocks
  (direct extension) and re-submit STARK txs from orphaned blocks
  (fork resolution)

The wire format is unchanged: newline-delimited JSON. STARK txs serialize
their ~33 KB proof as base64 inside the JSON payload, which works fine
but is wasteful (~33% size overhead). A real chain would use a binary
wire format.

### Stale-root behavior

A STARK proof attests against a *specific* Merkle root. When a STARK tx
arrives at a node whose pool has since moved on, `submit_stark_anon`
correctly rejects it with "stale Merkle root". The handler swallows this
silently — the network doesn't crash, the tx just doesn't enter the
mempool. The spender's wallet must rebuild the proof against the latest
root.

### The Gap D wrinkle

A subtlety surfaced by p2p propagation: **the STARK pool is not
currently chain-replicated.** Shielding to the STARK pool happens via
direct `chain.shield_to_stark_pool(leaf)` calls (e.g., from the
dashboard's `/api/stark/shield` endpoint), not via an on-chain
transaction.

This means two nodes can have *different STARK pools*. STARK txs only
verify on nodes whose pool root matches the proof's claimed root.

The candidate-chain rebuild in `_try_extend_or_replace` documents this:
it rebuilds the *nullifier set* from blocks (since spends are on-chain)
but **preserves** the existing `stark_anon_tree` (since shields aren't).

**Implication:** p2p of STARK *spends* between two nodes works as long
as those two nodes shielded the same notes. The new tests demonstrate
this by shielding the same note on both sides before testing the spend
gossip.

Real fix: add a `ShieldTransaction` type to the block. See
`M8.7-Gap-A-design.md` for that future work.

### Tests added

In `qchain/tests/test_network.py`:
- `test_stark_anon_tx_gossip_propagates` — both nodes pre-shield the
  same note; n1 spends; n2 receives via gossip with the full ~33 KB
  proof intact and re-verifiable.
- `test_stark_anon_tx_with_stale_pool_dropped_silently` — n1 has a pool
  with a leaf, n2 doesn't; n2 silently rejects the incoming tx without
  crashing.
- `test_stark_anon_tx_in_block_propagates` — n1 mines a block
  containing a STARK tx; n2 adopts the block and updates its
  `stark_nullifiers` set correctly.

## Part 2: Depth study

### What changed

`qstark/src/hash_air/m86_native.rs`:
- `M86_MERKLE_DEPTH` raised from 4 to **8**
- `M86_TRACE_LEN` now derives from depth via `const fn m86_trace_len_for_depth`
- `M86_NUM_BLOCKS` and `M86_ACTIVE_ROWS` derive from depth as before

`qstark/src/hash_air/m86_air.rs`:
- Fixed a latent bug: `ROOT_ROW` was hardcoded `5 * ROWS_PER_BLOCK - 1`.
  Now properly derived as `(M86_MERKLE_DEPTH + 1) * ROWS_PER_BLOCK - 1`.

`qstark_py/src/lib.rs`:
- Added `m86_merkle_depth()` Python binding (the old `merkle_depth()`
  exposed M8.3's constant, which is still 4).

`qchain/crypto/anon_stark.py`:
- Now uses `q.m86_merkle_depth()` — automatically follows whatever
  depth qstark is built with.

`qstark/tests/m86_bench.rs` (NEW):
- Measures proving time, verifying time, and proof size at the
  currently-compiled depth.
- Synthesizes a depth-correct witness without using the size-limited
  MerkleTree helper, so it works at any depth.

### Real measurements

Single-core, release-mode, on this benchmark machine:

| Depth | Anonymity set | Active rows | Trace len | Prove avg | Verify avg | Proof size |
|------:|---------------:|------------:|----------:|----------:|-----------:|-----------:|
|     4 | 16             | 48          | 64        | 3.25 ms   | 0.345 ms   | 28.3 KB    |
|   **8** | **256**          | **80**          | **128**       | **6.62 ms**   | **0.399 ms**   | **32.8 KB**    |
|    12 | 4096           | 112         | 128       | 5.92 ms*  | 0.313 ms*  | 33.3 KB*   |

*Depth 12 benchmarked with synthetic witness only — see "Test coverage cap" below.

### Why depth 8 was chosen

- All 16 M8.6 tests (smoke, soundness, diagnostic, native) pass at depth 8
- 256-note anonymity set is a real improvement over depth 4's 16
- Proving cost (~6.6 ms) is still comfortable for interactive UX
- Verifying time stays under 1 ms

Going to depth 12 (4096 notes) would be ~equal proving cost (same
padded trace length) but the qstark `MerkleTree` test helper is
hardcoded to `TREE_DEPTH = 8`, so the smoke/soundness tests would
need new infrastructure. A future depth-12+ milestone would:
1. Bump `TREE_DEPTH` in `merkle.rs` to 16 or so
2. Verify smoke + soundness pass at depth 12
3. Decide whether to ship depth 12 or higher

### Surprising scaling

The proving time **didn't scale linearly** with depth. From depth 4 to
depth 8, we went from 16× more notes (4× more Merkle blocks) but only
~2× proving time. This is because:

- Each new Merkle block adds 8 trace rows + reuses the same hash-round
  constraints (Rescue-Prime is the dominant work)
- The padded trace length grows only when crossing a power-of-2 boundary
  (64 → 128 going from depth 4 → 8)
- FRI work scales with trace length, not constraint count

This means a real production depth (20+) is plausible: proving
time would be ~3-4× depth 4, with anonymity set of millions.

### Test coverage cap

The qstark `MerkleTree` helper in `src/hash_air/merkle.rs` is currently
hardcoded to `TREE_DEPTH = 8`. The M8.6 smoke and soundness tests use
this helper to build the witness path, so they cap at `M86_MERKLE_DEPTH
≤ 8`.

The new `m86_bench` test uses a synthetic witness builder that supports
any depth, which is how we measured depth 12 above.

For future work: bump `TREE_DEPTH` so the test infrastructure tracks
the AIR depth. This is a separate refactor — not strictly required
for soundness, since the AIR itself is depth-parametric.

## What's next

Three honest options, in order of impact:

1. **Close Gap D** (~2 days) — add a `ShieldTransaction` type. The chain
   would then truly replicate STARK pools across nodes. This unlocks
   real multi-node STARK usage.

2. **Close Gap A** (~1 week) — value conservation via range proofs.
   See `M8.7-Gap-A-design.md` for the full design.
   This is the central upgrade for production-realism.

3. **Bump test infrastructure to depth 12+** (~half a day) — straightforward
   refactor of `merkle.rs`, then re-verify soundness at higher depths.

The natural sequence is (3) → (1) → (2). Doing (1) before (3) is fine
since (1) doesn't depend on test infrastructure.
