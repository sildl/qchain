## What Phase 4 verified

### Why Phase 4 is mostly verification, not new code

The network handler `_handle_new_stark_anon_tx` is a thin wrapper:
deserialize → `submit_stark_anon` → gossip. The defense IS the
existing `submit_stark_anon` path which already calls
`q.verify_m86_membership` with all five public inputs (root,
nullifier, unshield_amount, fee, **output_leaf**).

That means **Phase 3's chain integration automatically gave Phase 4
its tamper rejection for free** — the network handler delegates to
the same FS-protected verification. Phase 4 doesn't add new defense;
it adds tests proving the defense actually fires at the network
layer.

This is the right kind of finding. Layered defense should mean each
layer is independently verifiable, but the actual enforcement
typically happens at the deepest layer (the verifier). What network
tests prove is that the deeper defense REACHES the network surface
without being bypassed.

### Five new network adversarial tests

File: `qchain/tests/test_network.py` (appended)

| # | Test | What it proves |
|---|------|----------------|
| 1 | `test_m811_phase4_honest_partial_spend_propagates` | Real partial-spend tx flows between two nodes via gossip. output_leaf survives the wire roundtrip. After mining, both nodes' STARK pool roots converge. |
| 2 | `test_m811_phase4_malicious_peer_tampered_output_leaf_rejected` | Malicious node builds honest proof, tampers output_leaf post-construction, gossips directly (bypassing own validator). Honest peer rejects via FS cross-check. |
| 3 | `test_m811_phase4_tampered_output_leaf_rejected_consistently` | Same tampered tx, presented to two isolated nodes, rejected with same reason. Asymmetric rejection would indicate a stateful bug. |
| 4 | `test_m811_phase4_malformed_payload_missing_output_leaf_dropped` | Old-format payload missing output_leaf field is dropped cleanly by `from_dict`. Node doesn't crash, stays connectable. |
| 5 | `test_m811_phase4_full_spend_dummy_output_propagates` | Full-spend path (auto-generated dummy v_out=0 output_leaf) also propagates correctly. Backward-compat with the common case. |

All five passed on first run. No code changes were needed in the
network handlers — Phase 3's `STARKAnonTransaction.from_dict` and
`verify` updates already carried output_leaf through correctly.

### What test #2 (the headline test) actually demonstrates

The attack:
1. Malicious node M builds an honest partial-spend proof for
   `unshield=60, fee=0, v_out=40` (a real change note)
2. M tampers the tx's `output_leaf` field post-construction to be
   the hash of a DIFFERENT change note paying themselves more
3. M skips its own `submit_stark_anon_tx` (which would catch this)
   and gossips the payload directly to its peers

What honest peer P does:
1. Receives the payload via `_handle_new_stark_anon_tx`
2. Deserializes via `STARKAnonTransaction.from_dict` (gets the
   tampered output_leaf)
3. Calls `chain.submit_stark_anon(stx)` which calls `stx.verify()`
4. `verify()` calls `q.verify_m86_membership` with the tampered
   output_leaf as one of five public inputs
5. Fiat-Shamir transcript mismatch — proof was bound to the original
   output_leaf; the substituted one produces different FS challenges
6. Verification fails, `submit_stark_anon` raises ValueError
7. Handler catches the ValueError silently and drops the tx

Result: tampered tx never enters honest peer's mempool. The
malicious peer's attempt to steal value through output_leaf
substitution is rejected at the very first defense layer encountered.

This is the same Fiat-Shamir property that made M8.8-A1 Phase 4
work — extending it to the new M8.11 public input is automatic
because FS binds the full M86Inputs struct.

## What's still NOT done — honest scope

These remained out of scope across all four phases and remain so:

- **Wallet bookkeeping for change notes.** Spenders track their change
  notes via pool-scanning in tests. A `Wallet.shielded_notes` API would
  be the right abstraction but doesn't exist yet.
- **Dashboard UX for partial spends.** The "Spend STARK note" UI doesn't
  expose change-note fields. Python API supports it; UI doesn't.
- **Persistence.** Chain in-memory only. Restart loses everything,
  including the spender's knowledge of which leaves are their change
  notes. Same M1 limitation.
- **DoS hardening for the new gossip path.** A flood of malformed
  partial-spend txs would still consume CPU on every verify attempt.
  Network-layer rate limiting is Tier 1 production work.
- **Recursive STARKs / proof aggregation.** Each STARK tx adds ~0.6 ms
  of verify cost. At meaningful throughput this is a bottleneck.
  Real research target, not M8.11 scope.

These are documented in the existing ROADMAP.md as future work.
