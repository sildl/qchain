# Milestone 8.10 — STARK proof re-verification during chain replay

This milestone closes a hole flagged in M8.8-A1's "honest scope notes":

> `Blockchain.is_valid()` doesn't re-verify STARK proofs during chain
> replay (only signatures). Each STARK proof was verified at submission.

After M8.10, `is_valid()` re-verifies every STARK proof against the
historical STARK pool state.

## Why this matters

Before M8.10, the chain trusted that `submit_stark_anon` had caught any
malicious STARK tx at admission time. That trust was reasonable for
honest peers — but a malicious miner could include a forged proof in a
block via direct injection (bypassing their own admission check), and
honest peers replaying the chain via `is_valid()` would pass the chain
without ever checking the proof.

The attack scenario:
1. Malicious M generates a STARK proof for amount=100
2. M tampers `stx.unshield_amount = 50`
3. M injects the tx directly into a block they're mining (skipping their
   own submit_stark_anon)
4. M broadcasts the block
5. Honest peer H receives, runs admission checks on each tx in the
   block — `submit_stark_anon` rejects it
6. **But** if H processes the block through a fast path that calls
   `is_valid()` instead, the proof was never re-verified
7. H records an underspend in the ledger

In practice the current chain replay code in `_try_extend_or_replace`
DOES call individual admission paths, so the actual attack wasn't quite
this clean. But the hole was real and the comment in `is_valid()`
documented it openly. M8.10 closes it.

## What changed

### Single file: `qchain/chain/blockchain.py`

`is_valid()` now maintains parallel replay state for the STARK pool:

```python
replay_stark_tree = STARKAnonTree()
replay_stark_nullifiers: Set[StarkDigest] = set()
```

For each historical block N, before applying any state changes:

```python
for stx in curr.stark_anon_transactions:
    ok, reason = stx.verify(
        current_root=replay_stark_tree.root(),
        seen_nullifiers=replay_stark_nullifiers,
    )
    if not ok:
        return False
```

This mirrors exactly what `submit_stark_anon` does, except against the
replay state at the moment block N was mined rather than the current
chain state. The verify call invokes `qstark_py.verify_m86_membership`
internally, which checks the proof against (root, nullifier, amount,
fee) — all four bound via Fiat-Shamir.

After verifying all STARK txs in block N, state is applied in the
same order `mine_pending` uses:
1. M4 anon tx pool updates
2. Shield txs (extend STARK pool with new leaves)
3. STARK nullifier marks

The order matters: STARK spends in block N see the STARK pool as it
was BEFORE this block's shields, because that's the state they
committed to at submission time.

## What's NOT re-verified (honest scope)

- **Shield tx depositor balance.** The signature is re-checked, but the
  "did this depositor have enough coins?" check requires a full balance
  replay. `submit_shield` enforced it at admission; a production
  validator would replay balances too.
- **Smart contract execution** — there isn't any, but if there were,
  this'd be another replay vector.

## Tests added

4 new tests in `qchain/tests/test_anon_stark.py`:

| Test | What it proves |
|------|----------------|
| `test_m810_honest_stark_chain_validates` | Honest STARK-containing chain still passes `is_valid()` — proves the new code path actually runs and doesn't reject legitimate chains |
| `test_m810_tampered_stark_proof_in_historical_block_detected` | Flipping a byte in a STARK proof in a sealed block → `is_valid()` returns False |
| `test_m810_tampered_stark_amount_in_historical_block_detected` | Tampering `unshield_amount` in a sealed block → detected (the FS cross-check fires during replay too) |
| `test_m810_stark_tx_against_wrong_root_detected` | Tampering `merkle_root` in a sealed block → detected |

The honest-chain test also surfaced a useful design point: the
`fresh_chain_with_pool` test helper uses `chain.shield_to_stark_pool()`
directly, bypassing the shield-tx flow. That's a test shortcut that
violates the chain-replay invariant. The new `_build_chain_with_stark_txs`
helper goes through the real path (signed `ShieldTransaction`s mined into
blocks), which is the only path `is_valid()` can replay correctly.

## Performance

Measured on this machine:

| Chain shape | `is_valid()` time |
|-------------|------------------:|
| Empty (2 blocks: genesis + 1 shield) | 676 ms |
| Small (13 blocks, 1 STARK tx) | 695 ms |
| Medium (58 blocks, 5 STARK txs) | 689 ms |
| Larger (117 blocks, 10 STARK txs) | 692 ms |

**The cost is flat at ~700 ms regardless of chain length.** Surprising
finding from this milestone: the dominant cost in `is_valid()` is NOT
STARK verification but `STARKAnonTree.__init__` precomputing 65,535
empty-subtree digests for the depth-16 tree. Each STARK verification
adds only ~0.6 ms, so 1000 STARK txs in a chain add ~0.6 s.

The empty-tree precomputation cost has been present since M8.7-C; M8.10
just makes it more visible because is_valid() now allocates a second
STARKAnonTree (replay_stark_tree).

**Future optimization (M8.10.1):** Cache empty-subtree digests as a
class-level attribute in `STARKAnonTree`. Each tree depth has only ONE
possible empty digest per level; compute once across all instances.
That'd reduce is_valid() to milliseconds for empty chains and scale
linearly with actual STARK content. Out of scope for M8.10; documented
as a follow-up.
